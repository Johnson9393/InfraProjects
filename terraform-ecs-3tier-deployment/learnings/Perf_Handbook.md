# 1. What is Performance Testing?

Performance Testing is the process of measuring **how fast, stable and scalable** an application behaves when multiple users access it simultaneously.

Performance testing answers questions like:

- Can my application handle 100 users?
- Can it handle 1000 users?
- Does response time increase?
- Does CPU become 100%?
- Does Database become slow?
- Which component becomes the bottleneck?

Unlike Functional Testing, Performance Testing is **not checking whether the application works**, but **how well it works under load.**

# 20. Key Takeaways

✔ Performance Testing measures Speed, Stability and Scalability.

✔ Functional Testing verifies correctness.

✔ Virtual Users simulate real users.

✔ Response Time is the complete time taken to process a request.

✔ Throughput measures how much work the application completes.

✔ Error Rate indicates application failures.

✔ Database is usually the first bottleneck.

✔ Connection Pool reuses database connections instead of creating new ones.

✔ N+1 Queries significantly reduce application performance.

✔ Optimize application before scaling infrastructure.

✔ Performance Engineering is about identifying the actual bottleneck, not making assumptions.

# 45. Why Percentiles Exist

Percentiles tell us **how the majority of users experienced the application**, instead of averaging everyone's response time.

They help identify slow users that the average response time hides.

```

```

Checks verify **correctness**, not performance.

Example

```

```

✓ Check Passed
```

```

✗ Check Failed
```

# 56. Why Checks are Important

Suppose

```
1000 Requests

↓

All completed in 20 ms
```

Looks excellent.

But every request returned

```
HTTP 500
```

Without Checks,

you may wrongly conclude that the application is fast.

Checks ensure that every request returns the expected result before analyzing performance.

- Response Body
- Authentication Success
- Expected JSON Fields
- Business Validation

# 57. Thresholds

Thresholds validate whether the **entire performance test** met the expected performance objectives.

Unlike Checks,

Thresholds evaluate overall metrics such as

- P95
- Error Rate
- Throughput

Example

Requirement

```
P95 < 500 ms
```

Actual

```
P95 = 720 ms
```

Result

```
Threshold Failed
```

Thresholds automatically determine whether the load test passes or fails.

---

# 58. Checks vs Thresholds

| Checks | Thresholds |
|---------|------------|
| Validate individual requests | Validate entire performance test |
| Functional Validation | Performance Validation |
| Executed for every request | Evaluated using overall metrics |
| Example: Status Code = 200 | Example: P95 < 500 ms |

Simple way to remember

```
Checks

↓

Did this request work?
```

```
Thresholds

↓

Did the application meet the performance objective?
```

Useful for testing application stability.

---

### Ramping VUs

Gradually increases or decreases users.

Example

```
20

↓

100

↓

300

↓

500

↓

100
```

This is the most commonly used executor because it closely resembles real production traffic.

---

### Shared Iterations

Executes a fixed number of iterations regardless of the number of Virtual Users.

Example

```
Execute

1000 Requests

↓

Stop
```

Useful for batch processing tests.

---

### Constant Arrival Rate

Maintains a fixed request rate.

Example

```
200 Requests

Every Second
```

Useful for API throughput testing.

# 68. Initialization Phase

The **Initialization Phase** is the first stage executed by k6.

Everything outside any function executes **only once**.

Typical activities

- Load Configuration
- Define Test Options
- Read Test Files
- Declare Global Variables

Example

↓

↓

Define Test Duration
```

Think of Initialization as

```
Preparing the Exam Hall

↓

Before Students Enter
```

No requests are sent to the application during this phase.

---

# 69. setup()

The **setup()** function executes **only once** before any Virtual User starts.

Purpose

Prepare common resources required for the test.

Typical activities

- Generate Authentication Token
- Read CSV Files
- Create Test Data
- Fetch Configuration
- Create Temporary Resources

Example

```
Login Once

↓

Generate JWT Token

↓

Return Token

↓

Every Virtual User Uses Same Token
```

Instead of

```
100 Users

↓

100 Login Requests
```

we perform

```
1 Login

↓

Reuse Token
```

This reduces unnecessary traffic.

---

# 70. Virtual Users (VUs)

After setup(),

k6 creates Virtual Users.

Each Virtual User behaves like an independent user.

Every Virtual User has

- Independent Execution
- Independent Memory
- Independent Iterations

Virtual Users do **not** share local variables.

---

# 71. default()

The **default()** function is the heart of every k6 script.

Everything inside this function is executed repeatedly by every Virtual User.

Example

```
Login

↓

Dashboard

↓

Search

↓

Logout
```

Every Virtual User repeats this business flow until the test duration ends.

This is where actual performance testing happens.

---

# 72. teardown()

The **teardown()** function executes **only once** after all Virtual Users complete execution.

Purpose

Clean up resources created during testing.

Typical activities

- Delete Test Data
- Remove Temporary Resources
- Send Reports
- Notify Slack
- Close Sessions

Think of teardown() as

```
Cleaning the Exam Hall

↓

After Everyone Leaves
```

# 75. Common Beginner Mistake

Many beginners place Login inside **default()** even when Login is **not** the API being tested.

Result

```
500 Virtual Users

↓

500 Login Requests

↓

Every Iteration
```

This unintentionally creates unnecessary load on the Login API.

If Login performance is **not** the objective,

perform Login once in **setup()** and reuse the authentication token.

However,

if the Login API itself is being performance tested,

Login should remain inside **default()** because it is part of the workload being measured.
